# 需求闭环完成度检查清单

说明：本文件是闭环完成度、`CL-xx` 检查项、`U/Q` 状态合法性与回填判定的唯一规则源。其他文件只引用本文件编号，不重复展开判定细则。

## 1. 判定输出

- `闭环完成`：阻塞项为 `0`，且同时满足：
    - 所有 `Q` 项均满足第7节的状态合法性约束；
    - 所有 `U` 项都有处理结论（`已编目->Qx` 或 `不编目（已澄清）`）；
    - 所有“影响闭环验收”的 `U` 项已编目进 `Q` 台账。
- `未闭环`：存在任一阻塞项，或存在违反第7节状态合法性约束的 `Q` 项，或存在“待处理”`U` 项，或存在应编目未编目的 `U` 项。

## 2. 检查结果分级

- `阻塞`：必须在本轮补齐，否则结论只能为 `未闭环`。
- `非阻塞`：可进入下一轮闭环，但必须记录风险与跟踪动作。

## 4. 非阻塞风险定义

- 文档措辞可优化但不影响实现判定。
- 非关键字段约束缺失但不影响主流程。
- 非功能目标未量化但已有最低要求。

## 5. 最小补充问题集规则

- 仅保留对“阻塞项清零”有直接贡献的问题。
- 按优先级排序：`业务中断风险 > 数据错误风险 > 一致性风险 > 可维护性风险`。
- 同类问题合并，避免重复提问。

## 6. 输出格式

- `R编号` 仅用于 `final-check` 的非阻塞风险输出，格式为 `R` + 递增数字，不回填正文。

1. 结论：`闭环完成/未闭环`
2. 阻塞项清单：`Q编号 + 对应检查项(CL-xx) + 影响范围 + 建议动作`
3. 非阻塞风险：`Q编号/R编号 + 对应检查项(CL-xx) + 风险说明 + 建议跟踪`
4. 最小补充问题集：列出全部需补齐问题，按影响优先级排序
5. 下一次复评条件：`补齐哪些项后可复评为闭环完成`
6. 本轮新增Q项清单：`Qx + 触发检查项(CL-xx) + 问题描述 + 影响范围 + 建议动作 + 当前状态`（可为空）

## 7. U/Q 台账规则

- `U` 项仅承载用户来源问题或建议；用户新增问题先进入 `U` 区，仅在问题指向 `C` 区时才需要关联 `C` 编码。
- 用户新增“建议”若改变当前版本规则、契约、边界或验收口径，按问题处理，并执行 `U -> Q` 编目判断。
- 仅影响闭环验收的用户问题必须编目为新 `Qx`；不影响闭环验收的用户问题可保留在 `U` 区，并标记 `不编目（已澄清）`。
- 模型在 `close-loop/final-check` 新发现的问题直接新增为 `Qx`，不得使用 `Ux` 暂存，也不得直接记为 `RESOLVED`。
- `Q` 编号递增且不重排。
- `U` 项编目状态仅允许：`待处理`、`已编目`、`不编目`。
- `U` 项处理结论仅允许：`已编目->Qx`、`不编目（已澄清）`、`待处理`；判定为“闭环完成”时不得存在 `待处理`。
- `OPEN`：问题尚未给出可落地回复，或回复不足以形成实现规则。
- `OPEN` 项必须具备 `目标版本 + 落地位置`。
- 用户回复问题时，仅更新 `回复内容`，不直接改变问题状态。
- `RESOLVED`：问题已转化为可测试规则，并已回填到正文落地位置。若问题指向 `C` 区或影响验收，还需同步回填契约/验收条目。
- `RESOLVED/DEFERRED` 项必须具备 `决议版本`。
- `DEFERRED`：问题明确延期到后续版本处理，当前版本不实现。
